# AGENTS.md

Repository guide for coding agents working in `i18n_editor`.

## Project Overview

- Stack: Wails v2 app (`Go` backend + `React/TypeScript` frontend).
- Backend entrypoint: `main.go`; main app logic: `app.go`.
- Frontend app: `frontend/src`.
- Wails-generated bindings: `frontend/wailsjs` (auto-generated, do not hand-edit).
- Primary domain: i18n FR/NL key alignment, diff generation, change application, and export.

## Setup Commands

Run from repo root unless noted.

```bash
go mod download
cd frontend && npm install
```

## Build / Run Commands

### Full app (Wails)

```bash
wails dev
wails build
```

### Backend only

```bash
go build ./...
```

### Frontend only

```bash
cd frontend
npm run dev
npm run build
npm run preview
```

## Test Commands

Current automated tests are Go tests (`*_test.go` in repo root).

### Run all tests

```bash
go test ./...
```

### Run tests in current package only

```bash
go test .
```

### Run a single test by exact name (important)

```bash
go test . -run '^TestCreateBackupAndRestore$'
go test . -run '^TestParseDiffToStandardChanges$'
```

### Run a subset by regex

```bash
go test . -run 'TestParseDiff|TestCreateBackup'
```

### Disable test cache when debugging

```bash
go test ./... -count=1
```

## Architecture Notes

- `App` methods in `app.go` are bound to the frontend through Wails.
- Frontend calls backend via `frontend/wailsjs/go/main/App.js` / `.d.ts` wrappers.
- i18n locale files: `frontend/src/i18n/locales/{en,fr,nl}.json`.
- UI screens live under `frontend/src/screens`.

## Code Style - Go

- Always keep code `gofmt`-formatted (tabs, standard Go spacing/import order).
- Import groups: standard library first, then third-party packages.
- Prefer small helpers for JSON/path transformations over duplicated logic.
- Use `camelCase` for unexported identifiers; `PascalCase` for exported types/methods.
- Keep exported Wails methods on `App` stable unless frontend bindings are updated.
- Return errors; do not panic for user/input/runtime failures.
- Wrap errors with context using `%w`:
  - `fmt.Errorf("failed to read file: %w", err)`
- Keep messages actionable and user-facing where surfaced to UI.
- Use `filepath.Join` + `filepath.FromSlash` for repo-relative file paths.
- Preserve deterministic behavior where order matters (sorting keys before output).

## Code Style - TypeScript / React

- TS is `strict`; avoid `any` unless absolutely necessary.
- Functional components + hooks are the default.
- Component files and exported components use `PascalCase`.
- Local vars/functions use `camelCase`.
- Keep side effects in `useEffect`; keep render paths pure.
- Handle async actions with `try/catch/finally` and explicit loading/error state.
- Reuse existing UI primitives in `frontend/src/components/ui`.
- Prefer existing path alias `@/*` for frontend src imports.
- Keep backend binding imports from `../../wailsjs/...` as generated conventions expect.
- Prefer existing i18n translation keys (`t('...')`) over hardcoded UI copy.

## JSON / i18n Conventions

- Locale JSON structure is nested-object based; leaf keys are compared by dot paths.
- FR and NL files are expected to remain key-aligned for i18n workflows.
- Keep key naming and nesting consistent across `en`, `fr`, `nl`.
- Avoid ad-hoc schema changes in standardized diff payloads.
- Preserve current standardized shape (`action`, `path`, `segments`, `key`, `values`, `context`).

## Error Handling Expectations

- Backend: fail fast on invalid input (empty fields, missing paths, invalid JSON).
- Backend: include operation context in errors; avoid opaque messages.
- Frontend: display backend error text in screen-level error areas.
- Avoid swallowing errors; log details when useful for debugging.

## File Safety / Generated Files

- Do not manually edit files in `frontend/wailsjs/` (generated by Wails).
- Do not commit build artifacts unless explicitly requested.
- Be careful with `test_files/` and manual fixtures; they are used for local flows.

## Change Workflow for Agents

1. Read nearby code and follow existing patterns before changing architecture.
2. Make minimal, targeted changes.
3. Run relevant checks:
   - Backend change: `go test ./...` (or at least targeted `go test . -run ...`).
   - Frontend change: `cd frontend && npm run build`.
4. If backend API signatures change, ensure Wails bindings are regenerated via `wails dev` or `wails build`.
5. Summarize what changed, what was validated, and any known gaps.
